# 内容研究撰写者

这项技能充当您的写作伙伴，帮助您进行研究、制定大纲、撰写草稿和精炼内容，同时保持您独特的声音和风格。

## 何时使用此技能

- 撰写博客文章、文章或通讯
- 创建教育内容或教程
- 撰写思想领导力文章
- 研究和撰写案例研究
- 制作带有来源的技术文档
- 使用适当的引用和参考文献进行写作
- 改进钩子和引言
- 在写作过程中获得逐段反馈

## 此技能的功能

1. **协作大纲**：帮助您将想法结构化为连贯的大纲
2. **研究协助**：查找相关信息并添加引用
3. **改进钩子**：增强您的开头以吸引注意力
4. **逐段反馈**：在您写作时审查每个部分
5. **保持声音**：保持您的写作风格和语气
6. **引用管理**：正确添加和格式化参考文献
7. **迭代精炼**：通过多次草稿帮助您改进

## 如何使用

### 设置您的写作环境

为您的文章创建一个专用文件夹：
```
mkdir ~/writing/my-article-title
cd ~/writing/my-article-title
```

创建您的草稿文件：
```
touch article-draft.md
```

从该目录打开 Claude Code 并开始写作。

## 指令

当用户请求写作协助时：

1. **理解写作项目**

   提出澄清问题：
   - 主题和主要论点是什么？
   - 目标受众是谁？
   - 希望的长度/格式是什么？
   - 您的目标是什么？（教育、说服、娱乐、解释）
   - 是否包含现有研究或来源？
   - 您的写作风格是什么？（正式、对话、技术）

2. **协作大纲**

   帮助构建内容：

   ```markdown
   # 文章大纲：[标题]
   
   ## 钩子
   - [开场白/故事/统计数据]
   - [为什么读者应该关心]
   
   ## 引言
   - 背景
   - 问题陈述
   - 本文涵盖的内容
   
   ## 主要部分
   
   ### 第1部分：[标题]
   - 关键点A
   - 关键点B
   - 例子/证据
   - [所需研究：特定主题]
   
   ### 第2部分：[标题]
   - 关键点C
   - 关键点D
   - 需要数据/引用
   
   ### 第3部分：[标题]
   - 关键点E
   - 反对观点
   - 解决方案
   
   ## 结论
   - 主要要点总结
   - 呼吁行动
   - 最后的想法
   
   ## 研究待办事项
   - [ ] 找到关于[主题]的数据
   - [ ] 获取[概念]的例子
   - [ ] 为[主张]找到引用
   ```
   
   **迭代大纲**：
   - 根据反馈进行调整
   - 确保逻辑流程
   - 识别研究差距
   - 标记需要深入研究的部分

3. **进行研究**

   当用户请求对某个主题进行研究时：

   - 搜索相关信息
   - 找到可靠的来源
   - 提取关键事实、引言和数据
   - 按请求的格式添加引用
   
   示例输出：
   ```markdown
   ## 研究：AI对生产力的影响
   
   关键发现：
   
   1. **生产力提升**：研究表明，内容创作任务节省了40%的时间 [1]
   
   2. **采用率**：67%的知识工作者每周使用AI工具 [2]
   
   3. **专家引言**：“AI增强人类创造力，而不是取代它” - 简妮·史密斯博士，麻省理工学院 [3]
   
   引用：
   [1] 麦肯锡全球研究院。 (2024)。 "生成式AI的经济潜力"
   [2] Stack Overflow开发者调查 (2024)
   [3] 史密斯，J. (2024)。麻省理工学院技术评论访谈
   
   已添加到大纲中的第2部分。
   ```

4. **改进钩子**

   当用户分享引言时，分析和加强：

   **当前钩子分析**：
   - 工作良好：[积极元素]
   - 可以更强：[改进区域]
   - 情感影响：[当前 vs. 潜在]
   
   **建议的替代方案**：
   
   选项1：[大胆的声明]
   > [示例]
   *为什么它有效：[解释]*
   
   选项2：[个人故事]
   > [示例]
   *为什么它有效：[解释]*
   
   选项3：[令人惊讶的数据]
   > [示例]
   *为什么它有效：[解释]*
   
   **吸引注意的问题**：
   - 是否激发好奇心？
   - 是否承诺价值？
   - 是否足够具体？
   - 是否与受众匹配？

5. **提供逐段反馈**

   当用户撰写每个部分时，审查以下内容：

   ```markdown
   # 反馈：[部分名称]
   
   ## 工作良好 ✓
   - [优点1]
   - [优点2]
   - [优点3]
   
   ## 改进建议
   
   ### 清晰度
   - [具体问题] → [建议的修复]
   - [复杂的句子] → [更简单的替代方案]
   
   ### 流程
   - [过渡问题] → [更好的连接]
   - [段落顺序] → [建议的重新排序]
   
   ### 证据
   - [需要支持的论点] → [添加引用或示例]
   - [笼统的陈述] → [使更具体]
   
   ### 风格
   - [语气不一致] → [更好地匹配您的声音]
   - [词汇选择] → [更强的替代方案]
   
   ## 具体的行编辑
   
   原始：
   > [草稿中的确切引言]
   
   建议：
   > [改进版本]
   
   为什么：[解释]
   
   ## 需要考虑的问题
   - [引人深思的问题1]
   - [引人深思的问题2]
   
   准备进入下一部分！
   ```

6. **保持作者声音**

   重要原则：

   - **学习他们的风格**：阅读现有的写作样本
   - **建议，而不是替代**：提供选项，而不是指令
   - **匹配语气**：正式、随意、技术、友好
   - **尊重选择**：如果他们更喜欢自己的版本，支持它
   - **增强，而不是取代**：使他们的写作更好，而不是不同
   
   定期询问：
   - “这听起来像您吗？”
   - “这是否是正确的语气？”
   - “我应该更/更少[正式/随意/技术]？”

7. **引用管理**

   根据用户偏好处理参考文献：

   **行内引用**：
   ```markdown
   研究表明，生产力提高了40%（麦肯锡，2024）。
   ```
   
   **编号引用**：
   ```markdown
   研究表明，生产力提高了40% [1]。
   
   [1] 麦肯锡全球研究院。 (2024)...
   ```
   
   **脚注风格**：
   ```markdown
   研究表明，生产力提高了40%^1
   
   ^1: 麦肯锡全球研究院。 (2024)...
   ```
   
   维护一个运行中的引用列表：
   ```markdown
   ## 参考文献
   
   1. 作者。 (年份)。 "标题"。出版物。
   2. 作者。 (年份)。 "标题"。出版物。
   ...
   ```

8. **最终审查和润色**

   当草稿完成时，提供全面的反馈：

   ```markdown
   # 完整草稿审查
   
   ## 总体评估
   
   **优点**：
   - [主要优点1]
   - [主要优点2]
   - [主要优点3]
   
   **影响**：[总体有效性评估]
   
   ## 结构 & 流程
   - [对组织的评论]
   - [过渡质量]
   - [节奏评估]
   
   ## 内容质量
   - [论点强度]
   - [证据充分性]
   - [例子有效性]
   
   ## 技术质量
   - 语法和机械：[评估]
   - 一致性：[评估]
   - 引用：[完整性检查]
   
   ## 可读性
   - 清晰度评分：[评估]
   - 句子多样性：[评估]
   - 段落长度：[评估]
   
   ## 最终润色建议
   
   1. **引言**：[具体的改进]
   2. **正文**：[具体的改进]
   3. **结论**：[具体的改进]
   4. **标题**：[如果需要，提供选项]
   
   ## 发布前的检查清单
   - [ ] 所有主张都有来源
   - [ ] 引用格式正确
   - [ ] 例子清晰
   - [ ] 过渡流畅
   - [ ] 呼吁行动存在
   - [ ] 校对错别字
   
   准备发布！🚀
   ```

### 示例3：改进钩子

**用户原始钩子**：
> “产品管理正在因为AI而改变。在这篇文章中，我将讨论AI对产品经理的一些影响。”

**改进选项**：

**选项1（数据驱动）**：
> “上个月，我让AI分析了500次客户访谈。它只用了30分钟，而不是3周。产品管理将永远不会一样。”

> “如果您可以与每个客户交谈，阅读每条评论，分析每份支持工单——所有这些都发生在您早上喝咖啡之前，会怎样？”

**选项3（故事）**：
> “莎拉花了两周时间构建了错误的功能。这不是因为她不理解她的用户，而是因为她无法快速处理数百次访谈以发现模式。”

## 写作工作流程

### 博客文章工作流程
1. 一起制定大纲
2. 研究关键点
3. 写引言 → 获取反馈
4. 写正文部分 → 每部分反馈
5. 写结论 → 最终审查
6. 精炼和编辑

### 通讯工作流程
1. 讨论钩子想法
2. 快速大纲（较短格式）
3. 一次会议内撰写
4. 审查清晰度和链接
5. 快速润色

### 技术教程工作流程
1. 制定大纲步骤
2. 写代码示例
3. 添加解释
4. 测试说明
5. 添加故障排除部分
6. 最终审查准确性

### 思想领导力工作流程
1. 灵感独特角度
2. 研究现有观点
3. 发展您的论点
4. 用强烈的观点写作
5. 添加支持证据
6. 撰写引人入胜的结论

## 专业技巧

1. **在VS Code中工作**：比网页版Claude更适合长篇写作
2. **一次一个部分**：逐步获取反馈
3. **单独保存研究**：保留一个research.md文件
4. **版本控制草稿**：article-v1.md、article-v2.md等
5. **朗读**：使用反馈来识别笨拙的句子
6. **设定截止日期**：“我今天想完成草稿”
7. **休息**：写作，获取反馈，暂停，修改

## 最佳实践

### 对于研究
- 在引用之前验证来源
- 尽可能使用最新数据
- 平衡不同观点
- 链接到原始来源

### 对于反馈
- 明确您想要什么：“这太技术了吗？”
- 分享您的担忧：“我担心这个部分拖沓”
- 提出问题：“这个流程合理吗？”
- 请求替代方案：“有其他解释这个的方法吗？”

### 对于声音
- 分享您的写作样本
- 指定语气偏好
- 指出匹配之处：“这听起来像我！”
- 标记不匹配：“太正式了，不符合我的风格”
