# 向 Move 代码质量检查器贡献

感谢您对贡献的兴趣！本文档提供了关于如何为此项目贡献的指南。

## 行为准则

本项目遵循行为准则。通过参与，您应遵守此准则。请向项目维护者报告不可接受的行为。

## 我该如何贡献？

### 报告错误

在创建错误报告之前，请检查现有问题。在创建错误报告时，包括以下内容：

- **清晰的标题和描述**
- **重现问题的步骤**
- **预期行为与实际行为**
- **展示问题的代码示例**
- **环境详细信息**（操作系统、Claude Code 版本、Move 版本）

### 提出改进建议

改进建议以 GitHub 问题形式跟踪。在创建改进建议时，包括以下内容：

- **清晰的用例** - 为什么这个改进有用？
- **详细描述** 提议的功能
- **示例** 说明其如何工作
- **您考虑过的其他方法**

### 提交拉取请求

1. **Fork** 仓库
2. **从 `main` 创建一个具有描述性名称的分支**：
   - `feature/add-new-check`
   - `fix/parsing-error`
   - `docs/improve-readme`
3. **按照我们的编码标准进行更改**
4. **彻底测试您的更改**
5. **提交** 具有清晰、描述性的消息（见下文）
6. **将更改推送到您的 fork**
7. **提交一个具有清晰描述的拉取请求**

## 提交信息指南

我们遵循传统的提交格式以保持清晰的 Git 历史：

```
<type>: <subject>

<body>

<footer>
```

### 类型

- **feat**: 新功能
- **fix**: 错误修复
- **docs**: 文档更改
- **refactor**: 代码重构
- **test**: 添加或更新测试
- **chore**: 维护任务

### 示例

```
feat: 添加参数排序验证

实现了对 Move 函数参数排序的检查：
首先检查对象，然后检查能力，接着是原语、Clock 和
TxContext 最后。包括来自 Move Book 的示例。

关闭 #42
```

```
fix: 处理模块名称解析中的边缘情况

带有下划线的模块名称被错误地标记。
更新了正则表达式模式以正确匹配有效标识符。
```

```
docs: 添加常见问题的示例

为在 Move 代码库中发现的 5 个最常见问题添加了前后代码示例。
```

## 开发设置

1. 克隆仓库：
   ```bash
   git clone https://github.com/YOUR_USERNAME/move-code-quality-skill.git
   cd move-code-quality-skill
   ```

2. 链接到 Claude Code 技能目录：
   ```bash
   ln -s $(pwd) ~/.claude/skills/move-code-quality
   ```

3. 使用 Claude Code 测试您的更改

## 添加新的检查

当添加新的代码质量检查时：

1. **参考 Move Book** - 包含源 URL
2. **提供示例** - 既有错误模式也有良好模式
3. **解释原因** - 不仅说明错误是什么，还要说明为什么它很重要
4. **彻底测试** - 包括测试用例
5. **更新文档** - 添加到 README 和 SKILL.md

## 风格指南

### Markdown

- 使用 ATX 样式标题（`#` 而不是 `===`）
- 段落每行一个句子
- 代码块必须指定语言
- 使用相对链接进行内部引用

### 代码示例

- 保持示例最小化和专注
- 包含解释问题的注释
- 显示错误和正确的版本
- 使用真实的 Move 代码模式

## 测试

在提交 PR 之前：

1. 使用各种 Move 包测试技能
2. 验证所有清单项是否正常工作
3. 确保示例准确无误
4. 检查文档是否最新

## 有问题？

- 关于贡献的问题
- 指南的澄清
- 潜在更改的讨论
