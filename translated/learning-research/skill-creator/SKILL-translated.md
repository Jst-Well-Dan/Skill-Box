# 技能创建者

本技能提供创建有效技能的指导。

## 关于技能

技能是模块化、自包含的包，通过提供专业知识、工作流程和工具来扩展Claude的功能。将它们视为特定领域或任务的“入职指南”——它们将Claude从通用代理转变为具有程序性知识的专门代理，而没有任何模型可以完全拥有这些知识。

### 技能提供的内容

1. 专业知识工作流程 - 特定领域的多步骤程序
2. 工具集成 - 与特定文件格式或API一起工作的说明
3. 领域专业知识 - 公司特定的知识、模式、业务逻辑
4. 打包资源 - 用于复杂和重复性任务的脚本、参考资料和资产

### 技能的组成

每个技能都包含一个必需的SKILL.md文件和可选的打包资源：

```
skill-name/
├── SKILL.md (必需)
│   ├── YAML 前置元数据 (必需)
│   │   ├── name: (必需)
│   │   └── description: (必需)
│   └── Markdown 指令 (必需)
└── 打包资源 (可选)
    ├── scripts/          - 可执行代码（Python/Bash等）
    ├── references/       - 需要按需加载到上下文中的文档
    └── assets/           - 用于输出的文件（模板、图标、字体等）
```

#### SKILL.md (必需)

**元数据质量：** YAML前置元数据中的`name`和`description`决定了Claude何时使用该技能。请具体说明技能的功能以及何时使用它。使用第三人称（例如，“当...时应该使用此技能”而不是“你应该使用此技能”），以明确技能的功能和使用场景。

#### 打包资源 (可选)

##### 脚本 (`scripts/`)

用于需要确定性可靠性和反复重写的任务的可执行代码（Python/Bash等）。

- **何时包含：** 当相同的代码被反复重写或需要确定性可靠性时
- **示例：** `scripts/rotate_pdf.py` 用于PDF旋转任务
- **优点：** 令牌效率高，确定性高，可能在不加载到上下文的情况下执行
- **注意：** 脚本可能仍然需要由Claude读取以进行修补或针对特定环境的调整

##### 参考资料 (`references/`)

旨在按需加载到上下文中以告知Claude的过程和思考的文档和参考资料。

- **何时包含：** 当Claude在处理时需要引用文档
- **示例：** `references/finance.md` 用于财务模式，`references/mnda.md` 用于公司NDA模板，`references/policies.md` 用于公司政策，`references/api_docs.md` 用于API规范
- **用例：** 数据库模式、API文档、领域知识、公司政策、详细的流程指南
- **优点：** 保持SKILL.md简洁，仅在Claude确定需要时加载
- **最佳实践：** 如果文件很大（>10k单词），请在SKILL.md中包含grep搜索模式
- **避免重复：** 信息应存在于SKILL.md或参考资料文件中，而不是两者都存在。除非信息确实是技能的核心，否则优先使用参考资料文件，以保持SKILL.md简洁，同时使信息易于发现，而不会占用上下文窗口。在SKILL.md中仅保留基本程序性指令和流程指南；将详细参考资料、模式和示例移动到参考资料文件中。

##### 资产 (`assets/`)

不打算加载到上下文中，而是用于Claude产生的输出中的文件。

- **何时包含：** 当技能需要用于最终输出的文件时
- **示例：** `assets/logo.png` 用于品牌资产，`assets/slides.pptx` 用于PowerPoint模板，`assets/frontend-template/` 用于HTML/React样板，`assets/font.ttf` 用于字体
- **用例：** 模板、图像、图标、样板代码、字体、用于复制或修改的示例文档
- **优点：** 将输出资源与文档分开，使Claude能够在不加载到上下文的情况下使用文件

### 渐进式披露设计原则

技能使用一个三级加载系统来有效地管理上下文：

1. **元数据（名称 + 描述）** - 总是存在于上下文中（约100个单词）
2. **SKILL.md 正文** - 当技能触发时（<5k单词）
3. **打包资源** - 根据Claude的需要（无限*）

*无限，因为脚本可以在不读取到上下文窗口的情况下执行。

## 技能创建过程

要创建技能，按顺序遵循“技能创建过程”，除非有明确的理由说明为什么可以跳过这些步骤。

### 第1步：通过具体示例理解技能

只有在技能的使用模式已经清楚地理解时，才跳过此步骤。即使在与现有技能一起工作时，这也仍然很有价值。

为了创建一个有效的技能，清楚地了解技能将如何被使用的具体示例。这种理解可以来自直接的用户示例或经过用户反馈验证的生成示例。

例如，当构建图像编辑技能时，相关问题包括：

- “图像编辑技能应该支持哪些功能？编辑、旋转，还有其他吗？”
- “你能给出一些使用此技能的示例吗？”
- “我想象用户会要求像‘从这张图片中移除红眼’或‘旋转这张图片’这样的东西。你还能想象其他使用此技能的方式吗？”
- “用户会说什么才能触发此技能？”

为了避免让用户感到不知所措，不要在一条消息中提出太多问题。从最重要的问题开始，根据需要跟进以提高效果。

当对技能应支持的功能有一个清晰的了解时，完成此步骤。

### 第2步：规划可重用技能内容

要将具体示例转换为有效的技能，通过以下方式分析每个示例：

1. 考虑如何从头开始执行示例
2. 确定在反复执行这些工作流程时，哪些脚本、参考资料和资产会有所帮助

示例：当构建处理查询“帮助我旋转这个PDF”的`pdf-editor`技能时，分析显示：

1. 旋转PDF需要每次都重写相同的代码
2. 一个存储在技能中的`scripts/rotate_pdf.py`脚本会有所帮助

示例：当设计处理查询“为我构建一个待办事项应用”或“为我构建一个跟踪我步数的仪表板”的`frontend-webapp-builder`技能时，分析显示：

1. 编写前端Web应用需要每次都编写相同的HTML/React样板
2. 一个包含样板HTML/React项目文件的`assets/hello-world/`模板将有助于存储在技能中

示例：当构建处理查询“今天有多少用户登录？”的`big-query`技能时，分析显示：

1. 查询BigQuery需要每次都重新发现表模式和关系
2. 一个记录表模式的`references/schema.md`文件将有助于存储在技能中

为了确定技能的内容，通过分析每个具体示例来创建一个包含以下内容的可重用资源的列表：脚本、参考资料和资产。

### 第3步：初始化技能

到此为止，是时候实际创建技能了。

只有在正在开发的技能已经存在且需要迭代或打包时，才跳过此步骤。在这种情况下，继续到下一步。

从零开始创建新技能时，始终运行`init_skill.py`脚本。该脚本方便地生成一个新的模板技能目录，该目录自动包含技能所需的一切，使技能创建过程更加高效和可靠。

用法：

```bash
scripts/init_skill.py <skill-name> --path <output-directory>
```

该脚本：

- 在指定的路径创建技能目录
- 生成一个包含正确前置元数据和TODO占位符的SKILL.md模板
- 创建示例资源目录：`scripts/`、`references/`和`assets/`
- 在每个目录中添加示例文件，可以自定义或删除

初始化后，根据需要自定义或删除生成的SKILL.md和示例文件。

### 第4步：编辑技能

在编辑（新创建的或现有的）技能时，请记住，技能是为另一个Claude实例使用的。专注于包含对Claude有益且非显而易见的信息。考虑哪些程序性知识、特定领域的细节或可重用资产可以帮助另一个Claude实例更有效地执行这些任务。

#### 从可重用技能内容开始

为了开始实施，从上面确定的以下可重用资源开始：`scripts/`、`references/`和`assets/`文件。请注意，此步骤可能需要用户输入。例如，在实现`brand-guidelines`技能时，用户可能需要提供要存储在`assets/`中的品牌资产或模板，或要存储在`references/`中的文档。

此外，删除不需要的示例文件和目录。初始化脚本在`scripts/`、`references/`和`assets/`中创建示例文件以演示结构，但大多数技能不需要所有这些。

#### 更新SKILL.md

**写作风格：** 使用**祈使/不定式形式**（动词优先指令），而不是第二人称。使用客观、指导性的语言（例如，“要完成X，做Y”而不是“你应该做X”或“如果你需要做X”），以保持AI消费的一致性和清晰性。

为了完成SKILL.md，回答以下问题：

1. 技能的目的是什么，用几句话概括？
2. 应该何时使用此技能？
3. 实际上，Claude应该如何使用此技能？上面开发的所有可重用技能内容都应被引用，以便Claude知道如何使用它们。

### 第5步：打包技能

一旦技能准备就绪，它应该被打包成一个可分发的zip文件，并与用户共享。打包过程会自动验证技能，以确保它满足所有要求：

```bash
scripts/package_skill.py <path/to/skill-folder>
```

可选输出目录指定：

```bash
scripts/package_skill.py <path/to/skill-folder> ./dist
```

打包脚本将：

1. **验证**技能，自动检查：
   - YAML前置元数据格式和必需字段
   - 技能命名约定和目录结构
   - 描述的完整性和质量
   - 文件组织和资源引用

2. **打包**技能，如果验证通过，将创建一个以技能命名的zip文件（例如，`my-skill.zip`），其中包含所有文件，并保持适当的目录结构以供分发。

如果验证失败，脚本将报告错误并退出，不创建包。修复任何验证错误，然后再次运行打包命令。

### 第6步：迭代

在测试技能后，用户可能会要求改进。通常这发生在使用技能后，对技能的表现有新的了解。

**迭代工作流程：**
1. 在实际任务中使用技能
2. 注意到挑战或低效之处
3. 确定如何更新SKILL.md或打包资源
4. 实施更改并再次测试
